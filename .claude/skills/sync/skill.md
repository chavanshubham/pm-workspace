Rebuild all aggregated index files in the PM workspace. This is the "update one, update all" engine.

NOW: !`date +%Y-%m-%dT%H:%M`
TODAY: !`date +%Y-%m-%d`

## Steps

### Phase 1 — Discover all notes

1. Glob all `.md` files from these directories only (never touch `_index/` or `_templates/`):
   - `$WORKSPACE/daily/*.md`
   - `$WORKSPACE/meetings/*.md`
   - `$WORKSPACE/projects/*.md`
   - `$WORKSPACE/people/*.md`

### Phase 2 — Parse frontmatter

2. For each file, read its YAML frontmatter (between the first pair of `---` delimiters). Extract:
   - `type`, `title`, `date`, `slug` (if present)
   - `projects` (list of slugs)
   - `people` (list of slugs)
   - `status` (for project notes)
   - `action_items` list of `{owner, task, due, status}` (for meeting notes)
   - `resolves` list of `{source_meeting, owner, task, outcome, note}` (for meeting notes)
   - `decisions` list of strings (for meeting notes)
   - `role`, `team` (for person notes)

3. Build a **resolution map** from all `resolves:` entries across all meeting notes:
   `resolution_map`: `{ "source_meeting_slug|owner|task" → { outcome, note, resolved_in_title, resolved_in_file } }`
   The key is a composite of source_meeting slug + owner + task text (normalized: lowercase, trimmed).
   This map is used in all later phases to determine whether an action item is open or closed.

4. Build three in-memory maps:
   - **project_map**: `{ project_slug → [ { file, title, type, date, people, action_items, resolves, decisions } ] }`
   - **person_map**: `{ person_slug → [ { file, title, type, date, projects, action_items, resolves } ] }`
   - **date_map**: `{ date_string → [ { file, title, type, people, projects, action_items, resolves, decisions } ] }`

5. **Resolve action item statuses**: For every action item across all meetings, check if a matching key exists in resolution_map. If it does, treat the item's effective status as the `outcome` value (done/partial/carried-over), regardless of the `status` field in its own frontmatter.

6. Track orphaned slugs: slugs in `projects:` with no matching `$WORKSPACE/projects/<slug>.md`, or in `people:` with no matching `$WORKSPACE/people/<slug>.md`.

### Phase 3 — Write project indexes

7. For each slug in project_map:
   a. Read `$WORKSPACE/projects/<slug>.md` if it exists — extract status, owner, target_date.
   b. Separate entries by type: project doc, meetings, daily notes.
   c. For action items: use resolved statuses from resolution_map.
      - **Open**: effective status is open (not in resolution_map, or outcome is `partial`/`carried-over`)
      - **Closed**: outcome is `done`
   d. Collect all unique person slugs; resolve display names from `$WORKSPACE/people/<slug>.md` where available.
   e. Write `$WORKSPACE/_index/by-project/<slug>.md`:

```markdown
---
generated: NOW

project: SLUG
---

```

    Prefix past-due items (due < TODAY) with ⚠️.
